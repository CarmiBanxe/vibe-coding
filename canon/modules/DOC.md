# DOC — Документный модуль

**Профиль:** DOC (активен во всех профилях)  
**Назначение:** Правила работы с файлами, документами, скриптами, архивами

---

## 1. Работа с файлами

### Создание
- Создавать только если файл действительно нужен — не плодить лишнего
- Перед созданием — проверить что файл не существует (`Glob` или `Read`)
- Имена файлов: snake_case для Python, kebab-case для sh/md/yaml
- Кодировка: UTF-8, LF (не CRLF)

### Редактирование
- **Всегда читать** файл перед изменением (`Read` tool)
- Предпочитать `Edit` (точечные правки) над `Write` (перезапись)
- Кумулятивное редактирование — добавлять/обновлять секции, не затирать историю
- Не добавлять комментарии/docstrings к коду, который не менялся

### Удаление
- Требует явного подтверждения
- Перед удалением — проверить что файл не используется
- Предпочитать `git rm` над прямым удалением

## 2. Документы и markdown

### Структура markdown-файлов
```markdown
# Заголовок (один H1 на файл)

**Метаданные:** версия, дата, ответственный

---

## Раздел 1
## Раздел 2
```

### Правила содержания
- Не создавать README.md и документацию без явного запроса
- Технические решения → ADR (Architecture Decision Record)
- Статус задач → MEMORY.md, не в отдельных файлах
- Версионирование документов: таблица в конце файла

## 3. Скрипты (bash)

### Стандарт скриптов
```bash
#!/bin/bash
set -euo pipefail   # всегда: exit on error, undefined vars, pipe failures

# Описание: что делает скрипт, где запускать
# Usage: bash scripts/SCRIPTNAME.sh

# Шаги нумеруются: Step 1, Step 2, ...
```

### Обязательные элементы
- `set -euo pipefail` — первая строка после shebang
- Описание одной строкой в комментарии
- Каждый шаг: `echo "=== Step N: Что делаем ==="` перед действием
- Финальная проверка результата
- Сообщение об успехе/ошибке

### Запрещено в скриптах
- Хардкодить пароли, API-ключи, токены
- Использовать `sudo` без объяснения
- Делать `rm -rf` без `|| true` guard и проверки пути

## 4. Архивы и бэкапы

- Бэкапы перед деструктивными операциями: `cp -r dir dir.bak.$(date +%Y%m%d)`
- ZIP для передачи аудиторам: `compliance_snapshot.py` (G-13)
- Не хранить архивы в репозитории — только в `/tmp` или на сервере

---

## 5. Синхронизация между проектами

Перенос контента между проектами — **только по явной команде** с указанием обоих.

```bash
# Правильно: явная команда с обоими проектами
rsync -av ~/vibe-coding/src/compliance/ ~/developer/compliance/

# Неправильно: молчаливое заимствование контекста из чужого проекта
```

---

## 6. MEMORY.md — протокол обновления

MEMORY.md — живой документ проекта. Правила:
- Обновлять после каждого значимого действия (не trivial правки)
- Кумулятивно: только добавлять/обновлять секции, не перезаписывать
- Структура: Date + Action + Result + Commit SHA (если применимо)
- Пушить на GitHub сразу после обновления
