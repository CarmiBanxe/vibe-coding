# VERIFICATION — Верификация и проверка

**Правила:** КАНОН 8  
**Назначение:** 5-шаговый протокол верификации кода, документов, решений

---

## КАНОН 8 — Протокол верификации (5 шагов)

```
FIXATION → ANALYSIS → DRAFT → VERIFY → FINALIZATION
```

---

### Шаг 1: FIXATION — Фиксация состояния

Перед изменением — зафиксировать текущее состояние:

```bash
# Для кода:
git status && git log --oneline -5

# Для файлов:
sha256sum [файл] > /tmp/[файл].before.sha256

# Для тестов:
pytest --no-cov -q 2>&1 | tail -5  # текущий baseline
```

**Цель:** знать точку возврата, если что-то пойдёт не так.

---

### Шаг 2: ANALYSIS — Анализ

Прочитать и понять перед изменением:

1. **Читать файл целиком** (не угадывать структуру)
2. **Найти все зависимости** (`grep -r "import ..."`)
3. **Проверить тесты** — есть ли покрытие для изменяемой части
4. **Оценить blast radius** — что ещё может сломаться

**Правило "трёх шагов назад":**
- Когда впервые появилась проблема?
- Что изменилось к тому моменту?
- Каков root cause (не симптом)?

---

### Шаг 3: DRAFT — Черновик

Подготовить изменение, не применяя:

1. Описать изменение одним предложением
2. Перечислить затронутые файлы
3. Указать риски и откат

```
ИЗМЕНЕНИЕ: [суть]
ФАЙЛЫ: [список]
РИСК: low/medium/high
ОТКАТ: [как вернуть, если сломается]
```

---

### Шаг 4: VERIFY — Проверка после изменения

После применения изменения:

```bash
# 1. Тесты
pytest --no-cov -q

# 2. Линтер / статический анализ
semgrep --config auto [файл]

# 3. Проверка импортов (Python)
python -c "from compliance.xxx import yyy; print('OK')"

# 4. SHA проверка (для критичных файлов)
sha256sum [файл]
```

**Порог принятия:**
- Все тесты зелёные (или не хуже baseline)
- Нет новых semgrep/snyk предупреждений уровня ERROR
- Импорты работают

---

### Шаг 5: FINALIZATION — Финализация

После успешной проверки:

1. `git add [файлы]`
2. `git commit -m "feat(scope): описание — N tests"`
3. `git push`
4. Обновить MEMORY.md
5. Если изменение архитектурное → создать ADR

---

## Уровни верификации (BANXE)

| Уровень | Когда | Инструменты |
|---------|-------|-------------|
| 1 — Мгновенный | При изменении файла | Semgrep |
| 2 — Pre-commit | Перед коммитом | Semgrep + Snyk + secrets |
| 3 — PR | При создании PR | CodeRabbit + CodeQL + Qodo |

## Верификация compliance-компонентов

Для KYC/AML/Payments кода — тесты ОБЯЗАТЕЛЬНЫ:

```python
# Минимум 20 тестов для новой функции (паттерн T-01..T-NN)
# TTL-тесты: time.sleep(1.1) для проверки TTL=1s
# Baseline: 747 тестов — не допускать регрессии
```

## SHA-256 проверка критичных файлов

```bash
# Создать контрольную сумму
sha256sum SOUL.md > SOUL.md.sha256

# Проверить целостность
sha256sum -c SOUL.md.sha256
```

Использовать для: SOUL.md, INVARIANTS.md, compliance_config.yaml, *.rego файлов.
