## КАНОН 3 — MEMORY.md как живая память

### Обязанность обновления

После КАЖДОГО значимого действия — обновить MEMORY.md и запушить.

**Значимые действия (требуют обновления):**
- Исправление бага
- Новая функция / модуль
- Диагностика проблемы (даже без фикса)
- Изменение конфигурации
- Архитектурное решение
- Найденный инвариант / зависимость

**Не требуют обновления:**
- Тривиальные правки (typo, форматирование)
- Read-only исследование
- Промежуточный шаг в процессе

### Правила редактирования MEMORY.md

- **Кумулятивно**: только добавлять / обновлять секции
- **Не перезаписывать** с нуля — история важна
- Структура записи: `Date + Action + Result + Commit SHA`
- Пушить сразу после обновления

### Шаблон записи в MEMORY.md

```markdown
### [YYYY-MM-DD] [Что сделано]

**Действие:** [краткое описание]
**Результат:** [что получилось / изменилось]
**Коммит:** `git-sha` (если применимо)
**Примечание:** [важный контекст, если есть]
```

## КАНОН 4 — Всё через скрипты

### Принцип

Каждое действие на сервере = единый bash-скрипт в `scripts/`.

**Пользователь запускает ТОЛЬКО:**
```bash
cd ~/vibe-coding && git pull && bash scripts/SCRIPTNAME.sh
```

Никаких отдельных команд пользователю. Без исключений.

### Структура скрипта

```bash
#!/bin/bash
set -euo pipefail

# Описание: что делает скрипт, где запускать
# Usage: bash scripts/SCRIPTNAME.sh

echo "=== Step 1: [Что делаем] ==="
# действие

echo "=== Step 2: [Что делаем] ==="
# действие

echo "=== ✅ Done: [итог] ==="
```

### Обязательные элементы скрипта:
- `set -euo pipefail` — выход при ошибке
- Нумерация шагов: `=== Step N: ===`
- Финальная проверка результата
- Сообщение об успехе / ошибке
- Git push внутри скрипта (пользователь не делает вручную)

### Запрещено в скриптах:
- Хардкодить пароли, API-ключи, токены
- `sudo` без объяснения в комментарии
- `rm -rf` без проверки пути и `|| true` guard

### Claude Code запускает сам vs создаёт скрипт:

**Claude Code выполняет напрямую** (локально):
- `git add`, `commit`, `push`
- `pytest`
- Создание/редактирование файлов
- `pip install` локально

**Создаёт скрипт** (для сервера):
- Любые SSH-команды на GMKtec
- Деплой, рестарт сервисов
- Синхронизация файлов между машинами
- Повторяемые operations (бэкапы, проверки)

## КАНОН 10 — Автоматический Git

### Обязательная последовательность после КАЖДОГО создания/изменения файлов:

```
1. git add -A
2. git commit -m "описание"
3. git push
4. Обновить MEMORY.md
5. Распространить изменения в связанные проекты
```

### Запрещено спрашивать пользователя:

| Нельзя спрашивать | Выполнять самостоятельно |
|------------------|--------------------------|
| "Хочешь запушить?" | git push |
| "Какой коммит-месседж?" | Составить самостоятельно |
| "Распространить изменения?" | rsync / copy |
| "Что дальше?" | Завершить задачу полностью |

### Формат коммит-сообщений (BANXE)

```
feat(scope): краткое описание — N tests

Детали реализации.
Closes: G-XX (ссылка на GAP, если применимо)

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

**Типы:** `feat`, `fix`, `docs`, `test`, `refactor`, `chore`, `ci`

### Определение "задача завершена"

Задача = ЗАВЕРШЕНА только когда:
- ✅ Файлы созданы / изменены
- ✅ `git add + commit + push` выполнен
- ✅ MEMORY.md обновлён и запушен
- ✅ Изменения распространены на все связанные проекты

Незавершённая задача = не задача.
