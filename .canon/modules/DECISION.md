# DECISION — Модуль принятия решений

**Профиль:** DECISION (активен во всех профилях)  
**Назначение:** Протоколы выбора варианта, стратегии, приоритизации

---

## 1. Принцип минимального вмешательства

Когда есть несколько вариантов решения — выбирать тот, который:
1. Наименее необратим
2. Наименее рискован для production
3. Требует минимально изменений в существующем коде
4. Проще объяснить пользователю

---

## 2. Decision Framework

### Шаблон для значимых решений

```
ПРОБЛЕМА: [одно предложение — что не так]
ВАРИАНТЫ:
  A) [вариант] — риск: [low/medium/high], откат: [возможен/нет]
  B) [вариант] — риск: [low/medium/high], откат: [возможен/нет]
РЕКОМЕНДУЮ: [A / B] — потому что [1-2 причины]
ЖДУ ПОДТВЕРЖДЕНИЯ: да/нет
```

---

## 3. Приоритизация задач

### Матрица приоритетов

| Критичность | Срочность | Действие |
|-------------|-----------|----------|
| HIGH | HIGH | Немедленно, все остальное — стоп |
| HIGH | LOW | Планировать на ближайший sprint |
| LOW | HIGH | Делегировать или quick fix |
| LOW | LOW | Backlog |

### Compliance-специфичные приоритеты (BANXE)

```
Критичность = HIGH если:
  - затронут I-21..I-25 (инварианты)
  - production data под угрозой
  - регуляторный дедлайн (FCA, EU AI Act)
  - emergency stop активирован

Критичность = MEDIUM если:
  - тест падает (регрессия)
  - GAP-REGISTER gap открыт
  - SLA breach (>80ms pre-tx gate)

Критичность = LOW если:
  - документация не обновлена
  - warning (не error) в логах
  - cosmetic issue
```

---

## 4. Правило "трёх шагов назад"

Перед тем как исправить ошибку — сделать три шага назад:

1. **Понять контекст** — когда впервые появилась проблема? Что изменилось?
2. **Найти root cause** — не симптом, а причину
3. **Оценить blast radius** — что ещё может сломаться?

Только после этого — fix.

---

## 5. Решения с долгосрочными последствиями

Для решений, влияющих на архитектуру (CLASS_D):
- Создать ADR (Architecture Decision Record) в `banxe-architecture/decisions/`
- Обсудить с CTIO Олегом перед реализацией
- Обновить GAP-REGISTER если появился новый gap

Шаблон ADR:
```markdown
# ADR-NNN: Заголовок решения

**Дата:** YYYY-MM-DD
**Статус:** PROPOSED | ACCEPTED | DEPRECATED
**Автор:** [кто предложил]

## Контекст
## Решение
## Последствия
## Альтернативы
```

---

## 6. Когда отказаться от задачи

AI должен остановиться и эскалировать пользователю если:
- Задача требует доступа к production data без backup
- Задача противоречит инварианту (I-21..I-25)
- Задача требует CLASS_B/C/D изменения без approver
- Задача затрагивает чужой проект (изоляция)
- Три попытки fix — ошибка не устранена

Остановиться ≠ провал. Остановиться вовремя = профессионализм.
